RESTfulインタフェースとは

RESTfulなインターフェイスとは、RESTの特徴を備えたルートの事を指す。
RESTの世界では、ネットワーク上のコンテンツ（リソース）を全て一意なURLで表現する。
これらのURLに対して、HTTPメソッドのGET(取得)、POST(作成)、PATCH(更新)、DELETE(削除)を使ってアクセスを行う。

RESTとは、なに(リソース)をどうする(HTTPメソッド)かを表現する考え方であると言える。
RESTfulなインターフェイスを利用する事で、より統一感があり、意味がつかみやすいURLを設計出来る。

（※HTTP POST,GET,PATCH,DELETEは、それぞれCRUD-Create,Read,Update,Deleteに相当するものと考えられる）

Railsでは、RESTfulなインターフェイスに沿ってルート設計するのが基本である。
Railsが提供するform_for'やurl_forやlink_to`などのビューヘルパーに関しても、RESTfulなインターフェイスを前提に機能設計されているため、RESTfulなインターフェイスの方がより自然に表現できるケースが多い。


RESTfulインターフェイスを定義するresourcesメソッド

RESTfulなインターフェイスを定義するには、routes.rbでresourcesメソッドを呼び出す。

リソース名

リソースとはCRUDの対象となるコンテンツであると考えられ、具体的にはモデルによって取得/編集する書籍情報(books)、ユーザ情報(users)、レビュー情報(reviews)などが挙げられる。

下記は、ユーザ情報(users)の取得/編集を意図したルート設計の例である。

この場合、下記のようなURLとアクションがマッピングされる。

URL アクション HTTPメソッド 役割
/users(.:format) index GET ユーザ一覧画面を生成
/users/:id(.:format) show GET 個別ユーザ詳細画面を生成
/users/new(.:format) new GET 新規ユーザ登録画面を生成
/users(.:format) create POST 新規ユーザ登録画面からの入力を受けて登録処理
/users/:id/edit(.:format) edit GET 既存ユーザ編集画面を生成
/users/:id(.:format) update PATCH/PUT 編集画面からの入力を受けて更新処理
/users/:id(.:format) destroy DELETE 一覧画面で選択されたデータを削除処理

アクションは全て、リソース名に対応するUsersControllerコントローラに属する。
また、resourcesメソッドは、ビューヘルパーlink_toなどで値用できるUrlヘルパーも自動生成する。

生成されるビューヘルパーxxx_pathとxxx_urlの違いは、「xxx_pathは相対パス」「xxx_urlは絶対URL」を返す点である。


単一のリソースを定義するresourceメソッド

resourcesメソッドが複数ののリソースを対象としたRESTfulインターフェイスを作成するのに対して、resourceメソッドは単一のリソースを対象としたRESTfulインターフェースを定義できる。

単一のリソースとは、アプリケーション設定のように、唯一のリソースを指す。
「/config/15」ではなく、「/cofig」のようなURLである。

リソース名

下記は、configリソースの登録とそれによって生成されるルート定義である。

URL アクション HTTPメソッド 役割
/config(.:format) show GET 設定情報画面を表示
/config/new(.:format) new GET 新規の設定登録画面を表示
/config(.:format) create POST 登録画面からの入力を受けて登録処理
/config/edit(.:format) edit GET 既存設定編集画面を表示
/config(.:format) update PATCH/PUT 編集画面からの入力を受けて更新処理
/config(.:format) destroy DELETE 指定された設定情報を削除処理

resourcesメソッドと比較すると、indexアクション（一覧）に対応するルートが定義されないのと、show,edit,deleteなどで:idパラメータを要求しない点が異なる。

注意点としては、resourceメソッドでもConfiggsControllerコントローラ（複数形）にマッピングされるという点である。


RESTfulインターフェイスのカスタマイズ

resources/resourceメソッドの各種オプションを活用する事によって、予め決められたマッピングルールを自由にカスタマイズする事が可能になる。
実際のアプリケーションにおいては、標準ルールだけで機能を実現する事は難しいため、主要なオプションは理解する必要がある。


ルートパラメータの制約条件constraintsオプション

resources/resourceメソッドで自動生成されるURLには、:idという名前のルートパラメータが含まれている。
例えば「/users/:id(fortmat)」というルート定義の場合は、「/users/108」のようなURLで:idパラメータに108という値を渡す事が出来る。

ルートパラメータには、デフォルトで任意の値を渡す事が出来るが、予め渡される値が分かっている場合は、パラメータ自体の値に制限を設けておくのが望ましい。

下記は、booksリソースのidパラメータに対して「1〜2桁の数値である事」という制限を設定した例である。

ルートパラメータに対する制約条件は、このようにconstrainsオプションで「パラメータ名: 正規表現パターン」の形式で指定する事ができる。

この状態で「~/books/108」のようなURLでアプリケーションにアクセスした場合、idパラメータの条件は2桁であるためbooksリソースにはマッチせず、結果「採用すべきルートが見つからない」というエラーが返される。

リクエストパラメータの妥当性は、モデル側でチェックするのが基本であるが、ルートで遮断してしまうことで、不正な値をより確実に遮断できる。
複数のリソース定義に対して、同一の制約条件を課したい場合には、ブロック形式で制約条件を記述する事も可能である。

コントローラクラス、Urlヘルパーの名前を修正するcontrollers、asオプション

resources/resourceメソッドは、デフォルトで指定されたリソース名をもとに対応するコントローラを決定し、Urlヘルパーを生成する。

階層構造を持ったリソースを表現するresourcesメソッドのネスト

アプリケーションの中でリソース同士が親子関係にあることは良くある事である。
例えば、booksリソースは配下にreviewsリソースを伴う。

こうした親子関係にあるリソースは、URLとしても表現できた方が直感的である。
Railsでは、こうした関係をresources/resourceメソッドのネストによって表現する事が可能である。

resources/resourceメソッドのネストは理論上いくらでも可能であるが、原則として2階層までにしておくのが無難である。
それ以上のネストは却って分かりにくくなるため、もはや直感的ではなくなる。
→「/books/12/reviews/5/images/1」など


リソースの「浅い」ネストを表現するshallowオプション

resources/resourceメソッドのネストはリソース同士の関係を表すには便利であるが、反面、URLが冗長になるデメリットがある。

例えば、id=10の書籍に属するレビューを表すために~/books/10/reviewsのようなURLは妥当であるが、
id=10の書籍に属するid=8のレビューを表すのに~/books/10/reviews/8と表すよりは
~/reviews/8と表せた方がシンプルである。
(レビューのid値が書籍のidに依らず一意な場合)

そこで利用できるのがshallowオプションである。

shallowオプションによって、:idパラメータ（レビューのid値）を受け取らないindex/new/createアクションでのみ、:book_idパラメータが付与され、それ以外のアクションではネストされない（浅い）URLが生成される。
（showアクションのURL例/reviews/:id）

また、scopeメソッドとshallowオプションを併用する事で、「浅い」URLにプレフィックスを付与する事も出来る。

（showアクションのURL例b/reviews/:id）

非RESTfulなルートの定義

RailsではRESTfulインターフェイスが基本だが、必ずしもRESTの思想に沿うような状況ばかりではない。
そのような場合は、必ずしもREDTfulインターフェイスを適用するのではなく、よりシンプルな非RESRfulなルートの利用を検討するべきである。


非RESTfulルートの定義matchメソッド

非RESTfulなルートを定義するには、matchメソッドを利用する。

pattern: URLパターン、verb: 許可するHTTPメソッド(複数指定も可)、opts: 動作オプション

まずは、matchメソッドの最も汎用的なパターンである。

↓
GET|POST|PATCH /:controller(/:action(/:id))(.:format) :controller#action

上記は、下記のようなURLにマッチする。

  • /books
  • /hello/list
  • /blog/details/20160204
  • /blog/details/20160204.json

()の部分は省略可能であるため、必須なのは「:controller」だけである。
なお、このような自由にアクションを呼び出せてしまう記述を「ワイルドコントローラ」と呼び、思わぬアクションで呼び出せてしまうことから、脆弱性の原因となるため、本番環境での利用には適さない。

viaオプションでは、ルートを許可するHTTPメソッドを指定する。
Rails3では任意オプションであったが、Rails4から必須となっている。

ルートの優先順位
routes.rbでルートを追加する際の記述位置は重要である。
ルートの優先順位は記述の順序通りになるため、汎用的な「:controller(:/action(:id)(.format)」のようなルートは出来るだけ最後に記述するべきである。
同じ理由から、rootメソッドはroutes.rbの末尾で定義するようにする。
