前回の記事では、コントローラ開発におけるリクエスト・レスポンスについて紹介しました。
今回は6章の後半、状態管理・フィルタ・Applicationコントローラについて書いていきます。


状態管理

状態管理とは、複数のページ（アクション）間で情報を維持するための仕組みの事を指す。
状態管理の必要性を理解するには、改めてHTTP(HyperTextTransfer Protocol)の制約を理解しておく必要がある。

HTTPは、クライアントからのリクエストに応じて、サーバーがレスポンスを返して終わり。
という、シンプルなプロトコルである。

つまり、同じクライアントから何度リクエストを送ってもサーバーはこれを同一のクライアントとは見なさず、HTTPはステートレス（状態を維持できない）プロトコルであるといえる。

しかし、大抵のアプリケーションにおいてはページを跨がって、ユーザを特定できる要件が存在する。
Railsでは、状態管理の機能を補うために下記のような機能を提供している。

状態 概要
クッキー ブラウザに保存される小さなテキスト情報（Rails以外の環境でも利用できる汎用的な状態管理の手段）
セッション クッキー、キャッシュ、データベースなどに状態情報を保存する仕組み（最も利用する状態管理の方法）
フラッシュ 現在と次のリクエストでのみ維持できる特殊なセッション情報

（※厳密にはクエリ情報や隠しフィールドも状態管理の一種だが、用途が限定的なため上記では挙げていない。）


クッキーを取得、設定するcookiesメソッド

クッキー(Cookie)とは、クライアント側に保存される簡易なテキストファイルの事です。
原則として、Webの世界ではサーバがクライアントにデータを書き込む事は許されていないが、クッキーだけが唯一の例外で、サーバがクライアントに対して一時的に情報を記録するには、まずクッキーを利用する必要がある。

クッキーを利用する事により、アプリケーションにおいて、ページを跨がりユーザの識別・クライアント単位の情報を管理できるようになる。

クッキーを用いたユーザ識別の流れ

  • クライアントからサーバへの初回リクエスト
  • レスポンスヘッダ経由でクッキーを発行
  • 2回目以降のリクエストにおいて、リクエストヘッダ経由でクッキー情報を送信
  • クッキーに基づいて生成された画面を出力

name: クッキー名、key: オプション名、value: 値

下記は、初回アクセスで入力したメールアドレスをクッキーに保存して、2回目以降のアクセスで復元する例である。

なお、cookiesメソッドに渡せるオプションは以下である。

オプション名 概要 設定値（例）
value クッキーの値 Rio
expires クッキーの有効期限 3.hours.from_now
domain クッキーが有効なドメイン web-deli.com
path クッキーが有効なパス /~wings/
secure trueの場合、暗号化通信での尾見クッキーを送信 true
httponly HTTPクッキーを有効にするか true

valueとexpirseは事実上、必須のオプションであり、expiresオプションが省略された場合は、ブラウザを閉じたタイミングでクッキーが破棄される。
httponlyオプションは、HTTP通信のみでアクセスできるHTTPクッキーを有効化する。
これによって、Javascriptからのクッキーアクセスを遮断する事が出来る。


永続化クッキー、暗号化クッキー

parmanent/encriptedメソッドを値要する事で、永続化クッキー/暗号化クッキーを生成できる。

永続化クッキーは、正確には有効期限が20年後に設定されるクッキーである。
expireオプションが設定された場合でも、permanent設定が優先される。


セッションを利用するsessionメソッド

セッションとは、ユーザ単位に情報を管理するための仕組みであり、昨今のフレームワークの多くが同様の機能を提供している。
Railsのセッションでは、デフォルトでクッキーに全ての情報を保存するため、標準の状態ではクッキーとほとんど違いはない。
セッションの設定を変更する事で、保存先（データストア）を変更できるという特徴がある。
ブラウザが開いている間、データを維持したいケースでは、まずセッションを優先して利用するのが良い。

下記は、セッション情報のデータストア例である。

保存先 概要
クッキー（CookieStore） クライアントのクッキーとしてセッションを保存（デフォルト）。高速に動作する反面、クライアントサイドへの保存のため、機密情報の保存には不向き。格納サイズも4KBに制限
キャッシュ（CacheStore） アプリケーションのキャッシュにセッションを保存。さほど重要ではない、短命なデータの保存に利用
データベース（ActiveRecordStore） Active Record経由でアクセスできるデータベースにセッションを保存。サーバ保存のため、機密情報にも有利

なお、Rails4ではChacheStoreを推奨しているため、ActiveRecordStoreは外部gem化されている。

セッションを用いたユーザ識別の流れ

  • クライアントからサーバへの初回リクエスト
  • レスポンスヘッダ経由でセッションID・セッションDATAを発行、セッションIDをクッキーに保存
  • 2回目以降のリクエストにおいて、リクエストヘッダ経由でセッションIDを送信、セッションDATAと照合
  • セッション情報に基づいて生成された画面を出力
    ※セッション管理のためのクッキーを「セッションクッキー」と呼ぶ

ここでポイントとなるのは、サーバサイドでセッションを管理する場合も、そのキーだけはクッキー経由でやり取りされるという点である。

フラッシュを利用するflashメソッド

リダイレクト処理の前後で、一時的にデータを保存したいようなケースではflashメソッドが活用できる。
例えば、データの登録/更新処理を行った際の結果をリダイレクト先において「〜の保存に成功しました」のように表示するケースが挙げられる。

ここまでで学んだ仕組みだけだと、実現が厄介である。
テンプレート変数は現在のアクションとテンプレートでしか維持できないため、利用できない。
セッションの場合は、リダイレクト先のページで不要になったセッションを自分で削除する必要がある。

フラッシュでは、現在と次のリクエストでのみデータを維持する仕組みを持っているため、簡単に実装が行える。
「次のリクエストで動的に削除される機能を持ったセッション」と解釈出来る。


フラッシュの基本

フラッシュは、その性質上リダイレクト命令（redirect_toメソッド）と併せて利用するケースが多い。

key: キー名、value: 値

上記ではキー名を:msgとしている。
予約キーであるnoticeやalertであれば、参照に際してもローカル変数のようにアクセスできるが、その他のキーの場合はflash[:msg]のように、flashメソッド経由でアクセスしなければならない。


フラッシュのその他のメソッド

flashメソッド経由で以下のようなメソッドにアクセスする事も出来る。

メソッド 概要
flash.now[:key] 現在のアクションでのみ有効なフラッシュを定義
flash.keep(:key) 指定されたフラッシュを次のアクションに持ち越す(:keyを省略した場合は全フラッシュが対象)
flash.discard(:key) 指定されたフラッシュをはき(:keyを省略した場合は全フラッシュが対象)

フィルタ

フィルタは、アクションメソッドの前後一方、あるいは前後双方に付随する共通的な処理を実行するための仕組みである。
たとえば、アクセスログや認証、アクセス制御といった機能をアクション個別に記述しなくて済むようになるため、コードがスマートになる。


アクションの事前・事後に処理を実行するbrfore、afterメソッド

アクションの直前直後で実行すべき処理は、before/afterフィルタに記述する。
なお、Rails3ではbefore_filter/after_filterであったが、Rails4でbefore_action/after_actionに改名された。

method: フィルタとして適用されるメソッド名

フィルタの適用範囲をカスタマイズする

フィルタを利用する上で、適用範囲を理解しておく事は重要である。
フィルタを特的のコントローラ/アクションに適用/除外する方法について紹介する。


フィルタの適用範囲をカスタマイズするonly、exceptオプション

before_action、after_actionメソッドに対して、only/exceptオプションを指定する事で、適用範囲を制限する事が出来る。

上記は、beforeフィルタをindex/index2アクションにのみ適用して、afterフォルタをindexアクションに対してのみ適用しない指定を行っている。

onlyオプションでは、指定されたアクションに対してのみフィルタを適用し、exceptオプションでは指定されたアクションを除いてフィルタを適用する。
最も、only/exceptオプションの利用は必要最小限に留めるべきである。
あまり複雑なフィルタ設定は、コードの可読性を著しく低下させ、デバッグ時にも問題（論理的）を見つけにくくする要因となるためである。

そういう意味では、コントローラも出来るだけ同じフィルタを適用できるような設計にすべきであり、逆に一部のアクションにしか適用できないようなフィルタには、そもそもそれがフィルタとすべき処理なのか再検討する必要がある。

アプリケーション共通の挙動を定義するApplicationコントローラー

ApplicationControllerは、全てのコントローラクラスの基底クラスであり、ルートコントローラとも呼べる。
原則として、Applicationコントローラには、直接呼び出せるアクションを実装するべきではなく、アプリケーションで共通の機能のみ定義するべきである。

例えば

  • 個別のコントローラから呼び出せるヘルパーメソッド
  • すべて（あるいはほとんど）のコントローラで利用するフィルタ
  • アプリケーション共通の設定

共通フィルタの定義-ログイン機能の実装

フィルタは派生コントローラでも呼び出せる性質を利用して、アプリケーション共通で適用すべきフィルタは、Applicationコントローラで実装するのが良い。

ログインページにおいてはログイン認証が必要ないため、skip_before_actionによって処理を無効にしている。

その他、特定のアクションでもログイン認証をスキップしたい場合には、下記のようにonlyオプションを併用する。


共通な例外処理をまとめる-resque_fromメソッド

例外の中には、アクション毎に処理するべきものも存在するが、アクションレベルで例外を吸収してしまう事で、本来発生すべき例外情報が開発者の目に届かなくなる可能性がある。

アクションレベルでは、出来るだけ例外は発生するに任せ、必要であればアプリケーションレベルで例外処理するのが望ましい。
アプリケーションレベルで例外を捕捉するには、resqueメソッドを利用する。

except: 捕捉する例外、resquer: 例外を処理するメソッド

下記は、resque_fromメソッドでActiveRecord::RecordNotFound例外を捕捉する例である。
例として、ActiveRecord::RecordNotFound例外は、books#showアクションで「~/books/10080」の用に存在しないidが指定された際に、レコードが見つからなかった事を通知するために発生する。

捕捉
本番環境のRailsアプリケーションでは、発生した例外の種類に応じてHTTPステータスが割り振られ、それぞれのステータスコードに応じたエラーページが表示される。
例えば、RoutingError（ルーティングに失敗）やUnknownAction（アクションが不明）であれば404 Not Found、Exception(一般例外)であれば、500 Internal Server Errorが発生する。
上記のような例外が発生した際の表示を変更したい場合は、対応するエラーページのpublic/400.htmlやpublic/500.htmlを修正すれば良い。


クロスサイトフォージェリ対策を行う-protect_from_forgencyメソッド

クロスサイトフォージェリ（CSRF:Cross-Site Request Forgeries）とは、サイトに攻撃用のコード（一般的にはJavascript）を仕込む事で、アクセスしてきたユーザに対して、意図しない操作を行わせる攻撃を指す。

ここは自分の解釈
例えば、特定の悪意あるページにアクセスした際、fantom.jsなどのブラウザ内で特定の手順を行わせるスクリプトを強制的に実行させられ、ユーザ認証が完了しているサービスなどにおいて、商品を購入されたりなどが手法として考えられる。

Rilasにおいて、CSRF対策を有効にしているのは、Applicationコントローラとレイアウトの下記部分になる。

あとは、個別ページでform_for/form_tagやlink_toなどのビューヘルパーで、HTTPメソッドのためのフォーム/リンクを生成するだけである。
これによって、アプリケーション側でトークンと呼ばれる証明書のようなランダムな文字列が生成され、フォームにも自動的に埋め込まれ、次回リクエスト時に初回リクエストのトークンと等しい場合のみ処理を実行する仕組みである。

上記のような仕組みを例供するRailsにおいて、CSRF対策を行う場合、開発者が留意すべきポイントは以下になる。

  • HTTP GETによるリンクでのデータ操作は行わない（特に削除）
  • データ操作のリクエストは、form_for/form_tag、kin_toなどのビューヘルパー経由で生成する
  • レイアウトを自分で作成する場合は、csrf_meta_tagsメソッドの呼び出しを忘れない。

共通ロジックをモジュールにまとめるconcernsフォルダ

アプリケーション共通、という程ではないが複数コントローラ/モデルで共通したロジックがある場合、これを管理する仕組みとしてRails4では、標準的な場所として下記ディレクトリを用意している。

  • app/controller/concerns
  • app/models/concerns

Rails4では、複数のコントローラ/モデルを跨がって利用するロジックは、モジュールとして切り出し、concernディレクトリに格納するのが基本である。

モジュールの基本的な構文は以下の通りである。

call_clazz: インクルード元のクラスメソッドを呼び出すためのコード
clazz: クラスメソッドの定義
instance: インスタンスメソッドの定義
